# brale-core 模型侧会话缓存需求文档（单轮决策）

## 1. 文档目的

将用户对 **brale-core** 的“模型侧会话缓存”需求进行完整、可执行地整理。
本文件仅描述需求，不包含实施计划、排期和任务拆分。

---

## 2. 背景与现状

### 2.1 当前调用方式（现状）
- 当前 brale-core 主要采用单次 `chat/completions` 请求模式。
- 该模式默认是“请求级无状态”，模型端不会自动记忆上一请求内容。
- 本地已存在缓存与落库机制（如 stage cache、事件落库），但这不等于模型端会话上下文。

### 2.2 当前痛点
- Agent 与对应 Provider 之间存在重复信息传递成本。
- 为让 Provider 具备上下文，常需重复携带较长文本/结构信息，导致 token 开销较高。
- 用户希望“模型服务端上下文复用”，而非本地再拼接冗长 prompt。

---

## 3. 目标（Goal）

在 **单轮单次决策** 场景下，实现“模型侧会话缓存”能力：
- Agent 与其对应 Provider 能共享同一会话上下文；
- 不同 stage（indicator / structure / mechanics）之间彼此隔离；
- 单轮结束后，会话缓存立即清除（或失效），不跨轮延续。

---

## 4. 范围（Scope）

### 4.1 In Scope（本次范围内）
1. 单轮决策内的会话上下文共享（仅同 stage 的 Agent ↔ Provider）。
2. stage 级隔离会话：
   - indicator 一条会话链
   - structure 一条会话链
   - mechanics 一条会话链
3. 轮次结束后会话清理。
4. 与现有决策流程兼容，不改变核心业务语义（分析、判定、执行判断逻辑保持一致）。

### 4.2 Out of Scope（本次范围外）
1. 多轮长期记忆（跨轮持续会话）。
2. 全局共享会话（所有 stage 混用同一会话）。
3. 引入复杂外部“知识记忆库”作为主链路依赖。
4. 对策略逻辑、风控规则本身的改造。

---

## 5. 需求清单（功能需求）

## R1. 单轮会话创建
- 系统在每个 symbol 的单轮决策开始时，能够创建该轮会话上下文容器。
- 该容器须支持 3 条独立会话链（indicator / structure / mechanics）。

## R2. Agent-Provider 同 stage 会话复用
- 在同一轮内，某 stage 的 Agent 调用后，Provider 调用应复用同一模型侧会话上下文。
- 示例：
  - `agent.indicator -> provider.indicator` 共享同一会话。
  - `agent.structure -> provider.structure` 共享另一会话。
  - `agent.mechanics -> provider.mechanics` 共享第三会话。

## R3. Stage 间隔离
- indicator/structure/mechanics 三条会话链必须隔离，不得互相读取上下文。
- 任一 stage 上下文泄漏到其他 stage 视为不满足需求。

## R4. 单轮结束清理
- 该轮处理完成后（无论成功、失败或中断），会话必须执行清理。
- 清理标准：后续轮次不得继续复用上一轮模型侧上下文。

## R5. 不依赖本地拼长 prompt 作为主要方案
- 主诉求是模型侧会话复用。
- 本地拼接长 user/system prompt 不作为本需求核心实现路径。

## R6. 降低重复传输负担
- 在满足输出质量的前提下，减少同轮 Agent->Provider 重复数据重传。
- 目标是降低冗余 token 消耗（注意不承诺绝对 token 线性下降）。

## R7. 与现有调用能力兼容
- 新能力不应破坏现有无状态调用链路。
- 需支持保留 fallback（例如仍可走原 chat/completions 无状态模式）。

---

## 6. 非功能需求（NFR）

## N1. 可观测性
- 需能观测到每轮每 stage 的会话生命周期：创建、复用、结束/清理。
- 日志中应包含足够排障信息（symbol、stage、round 标识、会话状态）。

## N2. 可靠性
- 会话创建失败时，系统应可回退到无状态调用，不中断核心业务流程。
- 失败不应导致死锁或长时间阻塞。

## N3. 安全性与隔离
- 会话标识、模型调用凭据需按当前安全规范处理。
- 禁止跨 symbol / 跨 stage 的上下文串用。

## N4. 性能与成本
- 不显著增加单轮决策总时延。
- 在同质输入下，尽量降低重复 token 传输成本。

---

## 7. 验收标准（Acceptance Criteria）

## AC1. 同 stage 共享成立
- 在同一轮内，`agent.indicator` 输出可被 `provider.indicator` 在模型侧会话中直接利用。
- 其他 stage 同理。

## AC2. 隔离成立
- 在测试中，indicator 会话内容不会出现在 structure/mechanics 的模型上下文中。

## AC3. 单轮清理成立
- 第 N 轮结束后，第 N+1 轮无法访问第 N 轮模型侧会话内容。

## AC4. 回退能力成立
- 关闭会话模式后，系统行为与当前无状态模式一致。

## AC5. 业务正确性不退化
- 在既有样例上，输出结构与解析成功率不低于当前基线（尤其严格 JSON 解析链路）。

---

## 8. 约束与假设

1. 模型供应端必须支持可复用会话/线程语义（或等价机制）。
2. 供应端会话上下文计费规则可能仍包含历史上下文读取成本。
3. 本需求关注“单轮会话复用 + 清理”，不追求长期记忆。

---

## 9. 用户原始诉求映射

- 诉求 A：Agent 和 Provider 在同一链路共享上下文（以指标分析为例）
  - 对应：R2、AC1
- 诉求 B：不同 stage 各有各的缓存，不要全局缓存
  - 对应：R3、AC2
- 诉求 C：单轮结束即清除
  - 对应：R4、AC3
- 诉求 D：希望模型侧缓存，不要本地重复追加长 prompt
  - 对应：R5
- 诉求 E：降低 token 重复传输
  - 对应：R6、N4

---

## 10. 术语定义

- 单轮（Round）：一次完整的单次决策流程。
- Stage：indicator / structure / mechanics。
- 模型侧会话缓存：由模型服务端维护的上下文状态，不等同于本地内存或数据库缓存。
- 本地缓存：应用本地暂存（内存/数据库）的请求或结果数据。
