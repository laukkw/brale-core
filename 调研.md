## 一、cinar/indicator v2 库概览

| 属性 | 值 |
|------|-----|
| **模块路径** | `github.com/cinar/indicator/v2` |
| **语言** | Go 1.22+, 纯 Go 零外部依赖 (核心库) |
| **License** | AGPLv3 (与 brale-core 兼容) / 可商业授权 |
| **指标数量** | 70+ (趋势/动量/波动率/成交量/估值) |
| **策略数量** | 35+ (含组合策略 + 装饰器) |
| **数据源** | Tiingo, FileSystem CSV, SQL, InMemory, 可扩展 Repository 接口 |
| **核心设计** | Channel-based streaming — 所有计算通过 `<-chan T` 流式进行 |

### 包结构

| 包 | 作用 |
|-----|------|
| `helper` | 通用 channel 管道原语 (Map, Filter, Operate, Duplicate, Pipe, CSV I/O) |
| `trend` | 趋势指标 (EMA, SMA, DEMA, TEMA, KAMA, HMA, MACD, STC 等 40+) |
| `momentum` | 动量指标 (RSI, Stochastic RSI, Ichimoku, Fisher Transform 等 16+) |
| `volatility` | 波动率指标 (ATR, Bollinger, SuperTrend, Donchian, Keltner 等 13+) |
| `volume` | 成交量指标 (OBV, MFI, VWAP, CMF, Force Index 等 12+) |
| `asset` | Repository 接口, Snapshot 模型, 数据源实现, 同步引擎 |
| `strategy` | Strategy 接口, Action 类型, 组合策略 (And/Or/Majority/Split) |
| `strategy/decorator` | 策略装饰器 (StopLoss, NoLoss, Inverse) |
| `backtest` | 回测引擎 + HTML 报告生成 |
| `mcp` | MCP Server (独立 go.mod, 用于 AI 工具集成) |

### 核心数据模型

```go
// 基础数据单元 (OHLCV)
type Snapshot struct {
    Date   time.Time
    Open   float64
    High   float64
    Low    float64
    Close  float64
    Volume float64
}

// 所有指标统一接口: channel in → channel out
type Ma[T helper.Number] interface {
    Compute(<-chan T) <-chan T
    IdlePeriod() int
    String() string
}

// 策略统一接口
type Strategy interface {
    Name() string
    Compute(snapshots <-chan *asset.Snapshot) <-chan Action
    Report(snapshots <-chan *asset.Snapshot) *helper.Report
}
```

### 数据管道原语 (helper 包)

| 函数 | 作用 |

| `Map[F, T]` | 逐元素转换 |
| `Filter[T]` | 按条件过滤 |
| `Operate[A, B, R]` | 双 channel zip 运算 |
| `Duplicate[T]` | Fan-out: 一个 channel 复制到 N 个输出 |

| `Add, Subtract, Multiply, Divide` | channel 算术运算 |
| `Change, ChangePercent` | 差分运算 |

## 二、brale-core 现状对比

### 2.1 当前指标体系

**依赖库**: `go-talib` (C 绑定, 需 CGO) + `talib-cdl-go` (K 线形态)

| 指标 | 参数 | Agent | 来源 |
|------|------|-------|------|
| EMA (fast/mid/slow) | 21, 50, 200 | Indicator | go-talib |
| RSI | 14 | Indicator + Trend | go-talib |
| MACD | 12, 26, 9 | Indicator | go-talib |
| ATR | 14 | Indicator + Trend | go-talib |
| OBV | — | Indicator | 手动计算 |
| Bollinger Bands | 20, 2σ | Structure | go-talib |
| K 线形态 (8种) | — | Structure | talib-cdl-go |
| CVD | — | Mechanics | 手动计算 |
| OI / Funding / Long-Short / Fear&Greed / Liquidations | — | Mechanics | Binance API |

**go-talib 实际调用 (5 个函数)**:
- `talib.Ema(closes, period)` — indicator_compress.go
- `talib.Rsi(closes, period)` — indicator_compress.go, trend_compress_structure.go
- `talib.Atr(highs, lows, closes, period)` — indicator_compress.go, trend_compress_structure.go
- `talib.Macd(closes, fast, slow, signal)` — indicator_compress.go
- `talib.BBands(closes, period, 2, 2, talib.SMA)` — trend_compress_structure.go

### 2.2 数据管道架构

```
Binance Futures API
    ↓ (REST 批量拉取)
Snapshot Fetcher (internal/snapshot/fetcher.go)
    ↓
MarketSnapshot {Klines, OI, Funding, LongShort, Liquidations, FearGreed}
    ↓
FeatureCompressor (internal/decision/features/compressor.go)
    ├→ IndicatorBuilder → IndicatorJSON    [per symbol, per interval]
    ├→ TrendBuilder     → TrendJSON        [per symbol, per interval]
    └→ MechanicsBuilder → MechanicsSnapshot [per symbol]
    ↓
CompressionResult
    ↓
LLM Agent Service (3 个 Agent 并行)
    ├→ Indicator Agent → IndicatorSummary {MovementScore, Confidence, ...}
    ├→ Structure Agent → StructureSummary {Regime, Bias, ...}
    └→ Mechanics Agent → MechanicsSummary {Crowding, RiskLevel, ...}
    ↓
Direction Consensus (加权证据聚合 + 共振奖励)
    ↓
Gate (质量门控) → Sieve (力学过滤) → Plan Builder → ExecutionPlan
```

### 2.3 共识机制 (已有)

```
权重: Structure=1.0, Indicator=0.7, Mechanics=0.5
有效权重: Wi = BaseWeight_i * Confidence_i
聚合: Score = Σ(Wi*Score_i) / ΣWi
共识度: Agreement = |Σ(Wi*sgn(Score_i))| / ΣWi
覆盖度: Coverage = ΣWi / ΣBaseWeight_i
信心 = Coverage * Agreement + ResonanceBonus (capped 12%)
门槛: |Score| >= 0.35 AND Confidence >= 0.52
```

### 2.4 Hard Guard (已有)

3 个独立检查, 仅在 IN_POSITION 状态:
1. **Stop Loss Hit**: 价格触及止损
2. **RSI Extreme**: Long RSI >= 75 / Short RSI <= 25
3. **Circuit Breaker**: 5 分钟反向波动 >= 1.5%

## 三、结论与边界

这次集成不应该理解成“把 cinar/indicator 整套搬进来”，而是把其中对 brale-core 真有帮助的那部分拆出来。当前最值得引入的是纯 Go 指标能力和少量可配置思路，不值得引入的是它的 channel 架构、repository 数据源层、策略框架和回测框架。

这份方案的主结论如下：

1. 立即做：`KAMA`、`STC`、`SuperTrend`、`MFI`
2. 延后评估：`Donchian`、`anchored VWAP`、Hard Guard 规则化
3. 明确不做：`MajorityStrategy`、channel streaming 改造、整套 strategy/backtest/repository 搬运

边界也要说清楚：

- 不新增新的运行时或服务
- 不改变现有 `Feature -> Agent -> Consensus -> Gate/Sieve -> Risk` 主链路
- 不把 one-shot LLM 调用改成多轮工具调用
- 不在本轮解决 session VWAP 的数据源问题

从 GitHub 上游仓库的公开信息看，这个项目在 2025-04-12 还有 “No super trend indicator?” 的 issue，在 2025-09-19 还有 `%B + MFI` 相关 PR，说明它的策略层和指标层还在继续变化。本轮借鉴重点应放在指标计算能力，不放在策略框架。

## 四、逐项建议与取舍

### 4.1 Indicator Agent

#### KAMA

建议纳入，而且不是简单多加一条均线。更合理的做法是让它承担现有 `ema_fast` 的职责，作为更平滑的快线锚点。原因很简单，当前 [indicator_compress.go](/home/lauk/work/go/brale-core/internal/decision/features/indicator_compress.go) 里的快中慢三条 EMA 主要是在给 LLM 提供“结构关系”，不是在做传统均线交叉交易。`KAMA` 在震荡里更稳，在趋势里又能跟上，比 `EMA21` 更适合这个位置。

#### STC

建议纳入。它和 RSI、MACD 不是简单重复，真正补的是趋势进入和趋势切换时的相位信息。对于 `Indicator Agent` 的 `movement_score`、`movement_confidence`、`noise` 这几个输出，`STC` 会比再加一条普通均线更有用。

#### GMMA

不建议首批纳入。GMMA 的问题不是不能算，而是最容易把输入 JSON 变成一堆“数值堆料”。如果以后要做，也应该输出压缩后的语义，比如：

- 短组是否一致上行
- 长组是否一致上行
- 短长组是扩张还是收敛

而不是把 12 条 EMA 原样交给 LLM。

### 4.2 Structure Agent

#### SuperTrend

这是首批里优先级最高的指标。当前结构层已经有 `structure_points`、`structure_candidates`、`break_summary`、`smc` 这些离散特征，但缺少一个“当前状态”足够硬的锚点。`SuperTrend` 正好补这个空位，一方面能直接输出方向状态，另一方面能给风险层一个很自然的 trailing level。

#### Donchian

值得做，但不建议和现有 `range_high/range_low` 直接叠加。当前 [trend_compress_structure.go](/home/lauk/work/go/brale-core/internal/decision/features/trend_compress_structure.go) 已经把最近窗口的高低点做成候选位，再加 Donchian 很容易变成重复信息。更合理的方向是二选一：

1. 用 Donchian 替换 `range_high/range_low`
2. 保留现有 range 逻辑，但把 Donchian 定义成 `breakout_boundary`

如果没有这种边界设计，新增 Donchian 只会让结构输入更臃肿。

### 4.3 Mechanics Agent

#### MFI

建议纳入，但在文档和字段命名上要小心。`MFI` 的数学本质还是 OHLCV 技术指标，不是 OI、Funding、Long/Short 这类衍生品微结构信号。为了不把层次弄乱，建议在 `Mechanics` 输出里单独归类成 `volume_price_flow` 一类，而不是直接和 `CVD/OI/Funding` 混成一锅。

#### VWAP

现在不建议直接做“标准 VWAP”。理由不是指标没价值，而是本地数据模型还不适合它。默认配置里主要是 `1h/4h/1d` 周期，加上固定 `kline_limit=300`，这套输入更适合做 rolling 近似，不适合做有 session 语义的日内 VWAP。真要做，建议优先级是：

1. `anchored VWAP`
2. `rolling VWAP`
3. `session VWAP`

其中最适合 brale-core 的其实是 `anchored VWAP`，可以锚到最近 swing high、swing low、break event，甚至未来锚到开仓点。

### 4.4 架构与策略层

#### Channel streaming

不建议现在做。当前数据入口是 REST 批量快照，不是实时事件流；指标计算也不是整条链路的瓶颈，LLM 调用才是。即便把压缩器改成 channel pipeline，末端为了给 Agent 组织完整 JSON，还是要回到批量收敛。复杂度上去了，收益很难落地。

#### MCP tool use

值得单独立项，但不应该和指标引入混在一轮里。MCP 解决的是“Agent 什么时候应该主动求值、主动取数”的问题，不是“本地该多几个指标字段”的问题。这件事如果要做，前提是 Agent 模式从 one-shot 变成多轮工具调用，再单独评估延迟、成本和稳定性。

#### Decorator 思路

这个思路值得借鉴，但不建议照搬代码。brale-core 当前的 Hard Guard 已经是独立节点，只是还不够规则化。更适合本地的方向是把已有 guard 条件整理成规则表，比如：

- `price_stop_hit`
- `rsi_extreme_exit`
- `short_horizon_circuit_breaker`

以后如果要加 `supertrend_flip_exit` 或 `mfi_exhaustion_exit`，也能顺着这个结构挂进去。

#### MajorityStrategy

不建议引入。当前 brale-core 的共识机制本质上是加权证据聚合，不是简单多数投票。把它换成或补上一层 `MajorityStrategy`，不会增强系统，只会把一套更细的证据系统拉回简单投票。

## 五、Prompt 与输入影响

### 5.1 现状

当前 LLM 的 `user prompt` 很薄，本质是把压缩后的 JSON 原样传进去，再附一个“决策窗口”字段。调用路径见 `internal/llm/app/llm_prompts.go`：

- Indicator: `Indicator 输入` + 原始 `IndicatorJSON`
- Structure: `Trend 输入` + 原始 `TrendJSON`
- Mechanics: `Mechanics 输入` + 原始 `MechanicsSnapshot`

这意味着，只要压缩器新增字段，`user prompt` 会自动包含这些新数据。

### 5.2 真正需要改的地方

真正该改的是默认 `system prompt`，因为它决定模型如何理解新字段。当前默认文案在 [internal/config/prompts.go](/home/lauk/work/go/brale-core/internal/config/prompts.go)。

如果只改压缩器、不改 prompt，会出现一个典型问题：模型虽然看到了 `KAMA`、`STC`、`SuperTrend`、`MFI`，但仍然会按旧的 EMA、MACD、RSI、结构点、OI/Funding 习惯去做判断，新字段很容易沦为“存在但不参与决策”的背景噪声。

### 5.3 建议的 prompt 改动范围

#### Agent 层

- `Agent Indicator`：补充 `KAMA/STC` 的优先级，明确它们可作为 `expansion`、`alignment`、`noise` 的证据
- `Agent Structure`：补充 `SuperTrend` 是结构状态和风险锚点的重要输入；如果以后纳入 Donchian，再把它定义为边界或 breakout 证据
- `Agent Mechanics`：如果纳入 `MFI`，要明确它是量价流向证据，不是单纯的衍生品拥挤度字段

#### Provider 层

- `Provider Indicator`：允许 `STC` 明显上拐/下拐提升 `momentum_expansion` 的判断权重
- `Provider Structure`：允许 `SuperTrend` 状态和 level 影响 `clear_structure` 与 `integrity`
- `Provider Mechanics`：如果引入 `MFI`，它可以影响 `signal_tag`，但不应单独决定 `liquidation_stress`

#### In-position Provider 层

- `In-position Indicator`：允许 `STC` 和 `KAMA` 作为 `divergence_detected`、`momentum_sustaining` 的辅助证据
- `In-position Structure`：允许 `SuperTrend flip` 或价格跌破 `SuperTrend level` 触发 `tighten` 倾向
- `In-position Mechanics`：允许 `MFI extreme + crowding mismatch` 作为 `crowding_reversal` 的辅助证据

#### Risk prompt

当前 risk prompt 不必改 schema，但如果结构锚点摘要里新增 `supertrend_level` 或 `supertrend_state`，建议同步在文案里点名，让 LLM 知道这些字段能作为止损、收紧和 TP 参考。

### 5.4 这轮不建议改的 prompt 内容

- Agent 输出 JSON schema
- Provider 输出 JSON schema
- Risk 输出 JSON schema

先把解释规则改对，比先扩 schema 更重要。

## 六、详细实施计划

### 6.1 方案总览

推荐走“两段式首批 + 一段式后续”的节奏：

1. 首批第一段：`KAMA + SuperTrend`
2. 首批第二段：`STC + MFI`
3. 后续段：`Donchian / anchored VWAP / Guard 规则化`

这样拆的原因很直接。`KAMA + SuperTrend` 都属于高价值、低歧义的增强项，最容易看出收益；`STC + MFI` 虽然也值得做，但会更明显地改变 LLM 对信号的解释方式，适合放在第二段，避免所有变量一次性改动。

### 6.2 首批第一段：KAMA + SuperTrend

#### 目标

- 用 `KAMA` 替代或并存于 `ema_fast`
- 给结构层新增 `SuperTrend state + level`
- 让风险锚点摘要可以消费 `SuperTrend level`

#### 预期改动

- `go.mod`
- `internal/decision/features/indicator_compress.go`
- `internal/decision/features/trend_compress.go`
- `internal/decision/features/trend_compress_structure.go`
- `internal/decision/risk_prompt_anchors.go`
- `internal/runtime/runtime_builder_compression.go`
- `internal/config/types.go`
- `configs/symbols/*.toml`
- `internal/config/prompts.go`
- 新增对应 `_test.go`

这是一个明显超过 8 个文件的改动，需要按专题拆 commit 或拆 patch。

#### 实现要点

1. `Indicator` 层新增 `kama_fast`，并保留旧 `ema_fast` 一段时间用于对比
2. `Trend` 层新增 `supertrend` 块，至少包含 `state`、`level`、`distance_to_price`
3. `risk_prompt_anchors` 在最近上下锚点之外，再纳入 `supertrend_level`
4. prompt 文案补充 `KAMA` 和 `SuperTrend` 的解读优先级

### 6.3 首批第二段：STC + MFI

#### 目标

- 给 `Indicator Agent` 增加更快的趋势切换证据
- 给 `Mechanics Agent` 增加量价流向证据

#### 预期改动

- `internal/decision/features/indicator_compress.go`
- `internal/decision/features/mechanics_compress.go`
- `internal/config/prompts.go`
- 新增对应 `_test.go`

#### 实现要点

1. `Indicator` 层新增 `stc` 块，至少包含 `current`、`last_n`、`state`
2. `Mechanics` 层新增 `mfi_by_interval` 或同类结构，避免和衍生品信号混成同级键
3. provider / in-position 文案补充 `STC` 和 `MFI` 的使用规则

### 6.4 后续段：Donchian、anchored VWAP、Guard 规则化

这段不放进首批一起做，避免需求扩散。建议按下面顺序继续：

1. 确定 Donchian 与 `range_high/range_low` 的替换关系
2. 确定 VWAP 采用 `anchored` 还是 `rolling`
3. 把 Hard Guard 整理成规则表

### 6.5 明确不在本计划内的工作

- channel streaming 改造
- MCP tool use
- MajorityStrategy
- cinar backtest 直接接入主链路
- 全量替换 `go-talib`

这些都可能有价值，但都不是这轮的最小正确集。

## 七、测试、验收与回滚

### 7.1 测试路径

#### Happy path

- 指标字段在 bars 足够时能稳定产出
- 新字段进入 LLM 输入后，旧字段消费逻辑不回归
- `SuperTrend level` 能进入结构锚点摘要
- `KAMA/STC/MFI` 不影响未启用指标的币种

#### Error path

- bars 不足时字段自动省略，不报错
- 某个 interval 缺失时仅跳过该 interval
- `Mechanics` 缺少 K 线时，其他衍生品字段仍正常输出

#### Edge cases

- 平量或零量，MFI 计算不崩
- 横盘行情，KAMA 不产生异常跳变
- 极短序列时不输出误导性的 `SuperTrend` 状态
- 多周期同时存在时，risk anchor 选择顺序稳定

### 7.2 验收标准

- `CompressionResult` 新字段完整、稳定、无 NaN/Inf
- Agent 输出 schema 无变化
- Provider 输出 schema 无变化
- Risk prompt 输入中能看到新增结构锚点
- token 增量在可接受范围内

### 7.3 回滚策略

这轮改动不涉及数据库 schema，也不涉及持久化数据迁移，回滚成本低。必要时可直接：

1. 移除新增 dependency
2. 删除新增字段
3. 恢复默认 prompt 文案
4. 保留已存在的 `go-talib` 路径

## 八、审阅重点

交给他人审阅时，建议重点看下面几件事：

1. `MFI` 是否应该放在 `Mechanics`，还是另起一个量价扩展块
2. `KAMA` 是替代 `ema_fast`，还是先并存一个版本周期
3. `SuperTrend` 是否进入 risk anchors，还是只停留在结构层
4. `Donchian` 是否替换现有 `range_high/range_low`
5. `VWAP` 后续是走 `anchored` 还是 `rolling`
6. prompt 是否只改解释规则，暂不扩 schema

## 九、风险提示

### License
cinar/indicator v2 为 AGPLv3。brale-core 本身也是 AGPL, license 兼容。如果未来 brale-core 改为闭源或以 SaaS 形式提供, 需要向 cinar 购买商业授权。

### 计算精度
cinar 是纯 Go 重实现, go-talib 是 C 库 ta-lib 的绑定。两者在 EMA/RSI/ATR 等指标上可能存在微小精度差异 (通常在 float64 第 10+ 位小数)。Phase 1 共存期间应对比验证。

### Token 消耗
每新增一个指标, Agent prompt 的 JSON payload 会增大。需要监控:
- Compression output 大小变化
- LLM token 消耗和成本变化
- Agent 输出质量是否因输入增多而下降
