** 什么是 Brale

Brale 是一个量化交易引擎，通过多个大语言模型智能体（LLM Agent）分析市场行情，并结合确定性的 Go 规则引擎来强制执行风控约束。每个决策周期经过六个阶段：

| 阶段 | 职责 |
|------+------|
| *观测（Observe）* | 采集 K 线、指标、市场结构、衍生品与情绪数据 |
| *决策（Decide）* | 三个独立智能体并行分析；共识规则筛选信号 |
| *规划（Plan）* | 生成风控计划 — 入场价、止损、止盈、仓位大小 |
| *执行（Execute）* | 通过 Freqtrade 下单 |
| *监控（Monitor）* | 实时持仓风险追踪与止损调整 |
| *对账（Reconcile）* | 通过 Webhook / 轮询同步 Brale 与交易所状态 |

** 设计理念

- *LLM 提议，代码决断* — 模型建议止盈 / 止损位；Go 代码负责校验价格合法性、杠杆上限和爆仓安全。模型可能产生幻觉，代码会将其拦截。
- *三条独立智能体车道（Agent Lane）* — 指标（Indicator）、结构（Structure）和力学（Mechanics）智能体各自独立运行。共识阈值防止单一模型的误判。
- *Provider 锚定因子（Grounding Anchors）* — 每个 Provider 不仅接收其 Agent 的摘要，还会收到代码提取的结构化锚定数据（跨时间框架对齐、结构突破、力学冲突），用于交叉验证，降低幻觉风险。
- *经验驱动的记忆系统* — 三层记忆系统（工作记忆 Working、情景记忆 Episodic、语义记忆 Semantic）将近期上下文、历史交易复盘和长期规则注入 LLM 提示词。
- *影子指标引擎（Shadow Indicator Engine）* — 默认使用纯 Go 指标引擎（=ta=），可选配影子引擎进行交叉校验，确保指标数据可信。差异检测通过 ~bralectl indicator diff~ 执行。
- *Gate Replay 回测* — 将历史决策通过当前规则集重放，零 LLM 调用开销，用于快速验证规则调优效果。
- *全自动调度* — 决策周期、监控和对账由内置调度层统一驱动；默认后端为 River，无需外部 cron。
- *可插拔执行层* — 当前对接 Freqtrade；接口已解耦，可替换其他执行后端。

** 快速开始

*前置条件：*
- Docker 与 Docker Compose（v2.20+）
- 本地开发：Go 1.25+
- 最低内存：4 GB（TimescaleDB + Freqtrade + Brale 运行时约需 2–3 GB）

- 网络：需能访问 Binance API（=api.binance.com=）与 LLM 端点

** E2E 测试

brale-core 包含一个 suite 化的 E2E 测试框架，在隔离的 Docker 栈中验证完整交易生命周期。

#+begin_src bash
make e2e-start                                     # 启动隔离的 E2E 栈
make e2e-test                                      # 快速回归（ctl suite）
make e2e-test E2E_SUITE=ctl,decision,reconcile     # 运行多个 suite
make e2e-test E2E_SUITE=ctl,decision,reconcile E2E_TELEGRAM_NOTIFY=1  # 结束后推送 Telegram 报告
make e2e-test E2E_SUITE=lifecycle-force-close       # 完整生命周期测试

make e2e-reset                                     # 清理所有 E2E 数据
#+end_src

E2E 栈使用独立端口（PostgreSQL 15432 / Freqtrade 18080 / Brale 19991），不影响主栈运行。
若设置 =E2E_TELEGRAM_NOTIFY=1=，=make e2e-test= 会默认读取 =.env= 中的 =NOTIFICATION_TELEGRAM_TOKEN= / =NOTIFICATION_TELEGRAM_CHAT_ID=，并在测试结束后推送一条摘要消息到 Telegram；也可通过 =E2E_TELEGRAM_TOKEN= / =E2E_TELEGRAM_CHAT_ID= 覆盖。

** LLM Token 用量参考

单次决策周期 = 所有 Agent + Provider + 风控 LLM 调用的 Token 总和：

| 配置 | 输入 / 周期 | 输出 / 周期 | 日估算 |
|------+-------------+-------------+--------|
| 1 品种 × 3 时间框架 × 1 小时周期 | 11 K–12 K | 1.1 K–1.2 K | ≈ 300 K |
| 1 品种 × 3 时间框架 × 15 分钟周期 | 11 K–12 K | 1.1 K–1.2 K | ≈ 1.1 M–1.2 M |

多品种时按比例线性增长。启用记忆系统后，每周期额外增加约 500–1 K Token（工作记忆 / 情景记忆 / 语义记忆上下文注入）。

** What Is Brale

Brale is a quantitative trading engine that uses multiple LLM Agents to analyze markets and
a deterministic Go rule engine to enforce risk constraints. Each decision cycle flows through
six stages:

| Stage | Role |
|-------+------|
| *Observe* | Collect candlesticks, indicators, market structure, derivatives and sentiment |
| *Decide* | Three independent Agents analyze in parallel; consensus rules filter signals |
| *Plan* | Generate risk plan — entry, stop-loss, take-profit, position size |
| *Execute* | Place orders via Freqtrade |
| *Monitor* | Real-time position risk tracking and stop adjustment |
| *Reconcile* | Sync state between Brale and the exchange via webhooks/polling |

** Design Principles

- *LLM proposes, code disposes* — models suggest TP/SL levels; Go code enforces price validity,
  leverage caps, and liquidation safety. The model may hallucinate; the code catches it.
- *Three independent Agent lanes* — indicator, structure, and mechanics Agents run separately.
  Consensus thresholds prevent single-model misjudgment.
- *Provider grounding anchors* — each Provider receives its Agent summary *plus* code-extracted
  structured anchors (cross-timeframe alignment, structure breaks, mechanics conflicts) for
  cross-verification, reducing hallucination risk.
- *Experience-driven memory* — three-layer memory system (working, episodic, semantic) feeds
  recent context, historical trade reflections, and long-term rules into LLM prompts.
- *Shadow indicator engine* — default pure-Go engine (=ta=) with optional shadow engine for
  cross-validation, ensuring indicator data integrity. Run ~bralectl indicator diff~ for diff checks.
- *Gate Replay backtesting* — replay historical decisions through the current rule set with zero
  LLM calls, enabling rapid rule-tuning validation.
- *Fully automated scheduling* — decision cycles, monitoring, and reconciliation run through the
  built-in scheduler layer (River by default). No external cron required.
- *Swappable execution* — currently wired to Freqtrade; the interface is decoupled.

#+begin_src bash
make e2e-start                                     # Start isolated E2E stack

** LLM Token Usage

Single decision cycle = all Agent + Provider + risk LLM calls combined:

| Configuration | Input / cycle | Output / cycle | Daily estimate |
|---------------+---------------+----------------+----------------|
| 1 symbol × 3 tf × 1 h interval | 11 K–12 K | 1.1 K–1.2 K | ≈ 300 K |
| 1 symbol × 3 tf × 15 min interval | 11 K–12 K | 1.1 K–1.2 K | ≈ 1.1 M–1.2 M |

Scale linearly for multiple symbols. With memory enabled, add ~500–1 K tokens/cycle for
working/episodic/semantic context injection.
